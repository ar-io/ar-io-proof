# ar.io Verification Stack — Governance

> **Status: ratified v1.0 (2026-06-10), amended v1.1 (2026-06-11 — standards layer re-homed to `ar-io-proof`).** Defines who decides, how, and on what cadence for the stack's standardization layer: the ratified specs ([`architecture.md`](architecture.md), [`envelope-spec.md`](envelope-spec.md), [`evidence-bundle.md`](evidence-bundle.md)) and the conformance corpus (`ar-io-proof/test-vectors/`). Created per decision **D2** of [`v1.2-coordination/README.md`](https://github.com/ar-io/ar-io-api-guard/blob/main/docs/stack/v1.2-coordination/README.md) ("yolo v1; BDFL = repo owner; async docs PR").

## 1. Decision authority

- **BDFL: Phil Mataras (@vilenarios)**, repo owner of `ar-io-agent`.
- The BDFL is the final decision-maker for: changes to any ratified spec, corpus changes (vector additions/regenerations), admitting a new profile to the registry ([`../envelope-spec.md` §4](envelope-spec.md)), and adding a language kernel (§3 below).
- Delegation is allowed per-decision (e.g. a sibling-repo maintainer signs off a profile's body→verdict mapping) but the merge that lands the change is the BDFL's.

## 2. Cadence

- **Async via docs PR. No meetings required.**
- A proposed change to a ratified spec is a PR against the spec file in `ar-io-agent`. Discussion happens on the PR; the BDFL merging it *is* the decision. A rejected change is a closed PR with a one-line reason.
- Additive (minor) changes need no waiting period. Breaking (major) changes and new kernels require the 30-day RFC below.

## 3. Kernel-count budget

- The budget is **3 language kernels: Go, Python, JS/TS** ([`architecture.md` §2](architecture.md)).
- A **4th kernel requires a written justification plus a 30-day RFC**: a PR adding `docs/stack/rfc-kernel-<lang>.md` stating the consumer that needs it, the maintenance owner, and the CI plan for corpus-gating it; open for comment ≥30 days before the BDFL decides.
- A non-language verifier (HSM, ZK circuit, contract) that consumes pre-canonicalized digests per [`architecture.md` §2](architecture.md) does not count against the budget.

## 4. Conformance-corpus governance

- The corpus is `ar-io-proof/test-vectors/`, generated by `tools/gen-vectors/` (same repo) — **generated, never hand-edited** (CORP1); owned and homed in `ar-io-proof` (CORP2, amended v1.1 — originally homed in `ar-io-agent`; moved so the contract is public alongside the reference verifier). Producer repos MAY retain a vendored byte-identical copy for their own CI gates (`ar-io-agent` does); the vendored copy is synced from here, never edited in place.
- **Corpus changes are minor-version bumps to the corpus tag** (`test-vectors-v1.x`): new vectors, new event types, new profiles. **Breaking changes** — any change to the bytes of an existing vector — **are major** (`test-vectors-v2.0`) and require the 30-day RFC.
- **Every conformant downstream pins a corpus tag** (vendored copy or submodule) and records the tag it pins. A kernel is conformant iff it reproduces its pinned corpus byte-for-byte (K3); upgrading the pin is the downstream's explicit act, never implicit.
- Each tag's contents are recorded in `test-vectors/CORPUS-<major>.md` (file list + per-file SHA-256), committed on the tagged commit.

## 5. Status banners (canonical wording)

Every spec doc carries exactly one of these states in its header blockquote:

| State | Canonical wording | Meaning |
|---|---|---|
| `proposed` | `**Status: proposed — not yet ratified.**` | An idea on the table. May be revised or withdrawn without notice; do not build against it. |
| `draft` | `**Status: draft — under review for ratification.**` | Content-complete and under active review. RFC-2119 keywords state the *intended* contract; they become binding at ratification. |
| `ratified` | `**Status: ratified vX.Y (YYYY-MM-DD).**` | Normative. Changes only via §2: additive = minor, breaking = major + 30-day RFC. Downstream consumers may build and pin against it. |
| `deprecated` | `**Status: deprecated (YYYY-MM-DD) — superseded by <doc> vX.**` | No longer the contract. Retained because permanently-anchored artifacts may still verify against it. |

Examples: `**Status: ratified v1.0 (2026-06-10).**` ([`architecture.md`](architecture.md)); `**Status: proposed — not yet ratified.**` ([`../distribution.md`](https://github.com/ar-io/ar-io-agent/blob/main/docs/distribution.md)).

## 6. Escalation — reporting a contract conflict

A downstream consumer (a kernel, a producer, a verifier, an auditor) that hits a conflict between a ratified contract and observed behavior:

1. **Open an issue in `ar-io-agent`** titled `contract conflict: <spec> §<section>`, with the smallest reproduction — ideally a corpus vector (or candidate vector) the implementations disagree on. A consumer without access to the (currently private) repo — e.g. an external auditor — escalates the same content to the BDFL directly, and the issue is filed on their behalf; this path widens when the repos go public per [`../distribution.md`](https://github.com/ar-io/ar-io-agent/blob/main/docs/distribution.md).
2. **Do not silently work around it.** A workaround that diverges from the corpus is a kernel fork — the worst outcome in the system ([`architecture.md` §9](architecture.md)).
3. The BDFL triages: **implementation bug** → fix the implementation, corpus unchanged ("the verifier is wrong — pin the bug in the verifier, not the vector"); **spec erratum, additive clarification** → minor spec + corpus bump; **spec error, byte-affecting** → major bump via the 30-day RFC. A ratified contract is never changed silently.
4. Mid-implementation ambiguity in an active coordination wave escalates to the owning lane first, then the BDFL ([`v1.2-coordination/README.md`](https://github.com/ar-io/ar-io-api-guard/blob/main/docs/stack/v1.2-coordination/README.md)).
